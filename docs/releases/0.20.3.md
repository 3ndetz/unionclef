## What's new

### Tungsten pathfinder: parallel node generation (5-6x faster)

Profiling revealed that `getChildren()` — the method that generates
candidate movement nodes — accounts for **95%+ of total search time**.
Each child node runs a full Minecraft physics simulation (`Agent.tick`),
and all children are independent of each other.

We split the work into two phases:
1. **Collect input combinations** (forward/back/left/right, yaw, sprint,
   jump) — cheap, no physics involved.
2. **Create nodes in parallel** via `parallelStream` — each thread
   simulates its own `Agent.tick` against the world.

The filter and openSet stages turned out to be negligible (0.1 ms),
so the entire bottleneck was in sequential child creation.

#### Benchmark results

Same route, same start position, measured with `;settings debugTime true`:

| Metric | Before | After | Change |
|---|---|---|---|
| 193 children | 8.8 ms | 1.7 ms | **5.2x faster** |
| 145 children | 6.3 ms | 1.1 ms | **5.7x faster** |
| 127 children | 5.9 ms | 0.9 ms | **6.6x faster** |
| Cost per child | 0.046 ms | 0.008 ms | **5.7x faster** |
| Late-search nodes (GC pressure) | 21.7 ms | ~1.7 ms | **~12x faster** |

Time breakdown per node expansion (after optimization):
```
getChildren   ~1.5 ms  (95%)   ← was 8-22 ms
filter        ~0.1 ms  ( 5%)
openSet       ~0.1 ms  (<1%)
```

#### Built-in profiler

New setting: `;settings debugTime true`

Logs per-node timing breakdown to stdout:
```
Tungsten [node#42] 1.7ms total | getChildren=1.6ms (193 raw) | filter=0.1ms (8 valid) | openSet+update=0.1ms
```

Also logs block-space search time and overall search summary.
